Best Practices Analyzer Core Library
The DownloadURL Attribute

This indicates the root URL name that should be used to look for a new version of the configuration or binaries.  This is only needed if an application is supporting auto-downloads.  The customizable UI supplied by the library does support this by default but it can be disabled (this is controlled by BPACustimizations.AllowAutoDownloads).

Example Configuration Fragment

<Configuration
   ConfigVersion="1.6.7.3"
   ConfigName="ExBPA Master Input File"
   DownloadURL="http://www.microsoft.com/exchange/code/ExBPA"
   ArticleURL="http://go.microsoft.com/fwlink/?linkid=30939&amp;l={0}&amp;v=ExBPA.{1}&amp;id={2}"
   ChangeDescription="The changes made in this version are minor modifications to the configuration metadata."
>

Parent Element

Attribute Description
Configuration This element contains some general meta-information about the input configuration or the output data.

Possible Values

Value Description
URL string For the customizable UI, this URL is expected to be the root of the download directory where the tool will check for updates.  To check for new binaries, it will use the following URL:

<DownloadURL>/<Version>/<Culture>/<AppName>.nextversion.xml

Where:

<DownloadURL> is the value of this attribute
<Version> should be major.minor for the application
<Culture> is the local culture of the ExBPA machine
<AppName> is the customized name of the application

The file is expected to have the following simple schema:

<NextVersion AppVersion="w.x.y.z">
   <Description>text explaining what is in this version</>
   <AutoMSI>url of new MSI package for auto download</>
   <ManualMSI>url of new MSI package for manual download</>
</NextVersion>
If this file exists and contains an AppVersion greater than the current version than it will consider a new version of the binaries available.  The value of the Description element will be displayed to the user so they know what they are getting.  The user will also be given a choice whether they want to auto-download the binaries (in which case after the download completes they will be told to exit and restart the application) or to manually download the binaries.  The above format allows for separate MSI packages to be used in both cases.

The customizable UI will also look for a new version of the configuration.  It will look for the following file:

<DownloadURL>/<Version>/<Culture>/<AppName>.config.xml

It will extract the ConfigVersion from this file and compare it to the local copy.

Note the UI also verifies that this URL name starts with "http://www.microsoft.com" to avoid possible spoofing issues.
<any> If a separate UI implementation is used, this can be any format desired (or not used).
