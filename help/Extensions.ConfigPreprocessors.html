Best Practices Analyzer Core Library
Configuration Preprocessors

Configuration preprocessors can be used to manipulate the configuration data before the engine processes it.  These are defined using ConfigPreprocessor elements and they will be executed in the order they are defined.  Each implementation can do whatever it likes to the configuration file as long as the result conforms to the BPA schema.

There are two built-in configuration preprocessor defined (the first doesn't actually use the extensible interface so it does not need to be defined in the configuration, but it gives you an idea of what can be done with this type of extension):

Restriction This limits the collection to run on only part of the configuration, based on which options are specified. The types and possible options of a restriction are defined at the top of the configuration file. Attributes can be specified on any element. In general, if the options specified match the attributes set on the node, the node, its ancestors, and its descendants are run (though additional restriction attributes may take precedence on descendants). Each type of restriction is independent, removing or filtering elements that do not match the options specified.
Exchange defines four restrictions: Category, Level, Scope, and Task. Category is used to control what sort of information is collected. Level controls the amount of detail, with the more-specific options including the less-specific ones. Scope limits a scan to a particular set of servers or admin groups. Unlike the others, it specifies instances to be filtered out during the collection. Finally, Task options include collections of other restriction options, but do not actually restrict the configuration themselves. There are three tasks defined for an Exchange configuration: a full health check, which does everything; a connectivity test, which does the minimum checks to see if a user has all the access they need; and a baseline scan which is a specialized comparison of settings against custom values.
Type This preprocessor will go through a configuration file and process Reference, Type and Customize elements.  When it is done, all the referenced types will be inserted properly into the configuration and the processed elements will be removed (since the core engine doesn't know or care about them).  Configuration information for the Type preprocessor:

<ConfigPreprocessor
   Name="0 Type Preprocessor"
   Assembly="BPA.Common.dll"
   Class=
"Microsoft.WindowsServerSystem.BestPracticesAnalyzer.Common.TypeConfigPreprocessor"
/>
